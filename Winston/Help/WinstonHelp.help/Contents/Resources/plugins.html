Plugins

Optional community add-ons that extend Winston.

Plugins are small add-ons — written in JavaScript — that can do things like fill in missing book details or generate reports. They are off by default: nothing runs until you install a plugin and switch it on, and each one can only do what you explicitly allow.

Are plugins safe?

  • A plugin runs in a sandbox with no direct access to your files, the network, or other apps. It can only use the specific abilities Winston offers.
  • Before a plugin runs for the first time, Winston shows you exactly what it wants to do (for example, “Read library metadata”) and asks you to confirm.
  • It can never overwrite metadata you've entered or edited — it may only fill in blanks.
  • Online look-ups still obey your Settings → Metadata switch, so a plugin makes no network calls while online metadata is off.
  • If a plugin misbehaves — errors repeatedly or hangs — Winston disables it automatically (“quarantines” it) without affecting the rest of the app.

Installing a plugin

  • Open Settings → Plugins and click Open Plugins Folder. (It lives at ~/Library/Application Support/Winston/Plugins/.)
  • Put the plugin's folder inside — each plugin is one folder containing a manifest.json file and a script.
  • Back in Settings → Plugins, click Refresh. The plugin appears in the list, Disabled.

Enabling & permissions

  • Flip the plugin's switch to turn it on.
  • Winston shows an Enable “…”? sheet listing the permissions it requests. Read them, then click Enable (or Cancel).
  • Common permissions are Read library metadata, Fill in missing book metadata, and Show notifications.
  • If a later version of a plugin asks for more, Winston asks you to confirm again — an update can't quietly gain new access.

Managing plugins

  • Each plugin shows a status: Active, Disabled, Invalid (the folder isn't a usable plugin), Failed (it couldn't start — the reason is shown), or Quarantined (disabled after repeated errors).
  • Expand Log to see the plugin's recent messages — handy for seeing what it did or why it failed.
  • Show in Finder reveals the plugin's folder.
  • Turn the switch off any time to disable a plugin; a quarantined one can be re-enabled to try again.

Troubleshooting

  • The plugin doesn't appear after I add it. Its folder name must match the id inside its manifest.json, and it needs a manifest.json plus the script. Click Refresh.
  • It says “Invalid”. The manifest is missing or malformed — the reason is shown next to the status.
  • It says “Failed”. The plugin's script hit an error on start-up; check the Log for details.
  • It says “Quarantined”. It errored too many times or took too long. The rest of Winston is unaffected; re-enable it to retry once its author fixes it.

Writing your own

  • Plugins are plain JavaScript that call a small Winston API — read the library, look up metadata, save a little state, and show notifications.
  • A ready-to-copy example and a step-by-step beginner tutorial (“Your First Metadata Fetcher”) ship with the Winston project under docs/example-plugin/ and docs/WritingPlugins.md; the complete API reference is docs/PluginAPI.md.
